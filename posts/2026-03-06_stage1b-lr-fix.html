1 Context

This report documents Stage 1b, a targeted fix for the learning rate instability diagnosed in Stage 1 (2026-03-03). Stage 1b uses the same architecture (unet_tiny, 6,849 params), same data (NIR/Blue/SWIR2), and same train/val split. The only changes are to the optimizer settings.

For full project background, data description, architecture details, and anticipated questions, see the Stage 1 report.

2 What Changed

3.1 Interpreting the Improvement

AUC: 0.930 → 0.976 (+0.046). This is a substantial improvement. In concrete terms: picking one random building pixel and one random non-building pixel, the model now ranks the building pixel higher 97.6% of the time (vs 93.0% before). The gain comes entirely from optimizer convergence — the model architecture and data are identical.

F1: 0.974 → 0.974 (unchanged). F1 was already near the Open Buildings label ceiling (~0.97–0.98) at Stage 1. The improvement in AUC without F1 change means the model is now more confidently discriminating buildings from non-buildings (better probability calibration), even though the binary classification outcome at the chosen threshold is essentially the same.

Threshold: 0.15 → 0.90. The large threshold shift is the most important qualitative change. In Stage 1, the model’s probability outputs were poorly calibrated — needing a very low threshold (0.15) to capture buildings. In Stage 1b, the model is far more confident: building pixels receive probabilities near 1.0 and non-building pixels near 0.0. A threshold of 0.90 achieves the same F1. This means the model’s probability map is now directly interpretable as a building density estimate — crucial for the Stage 2 aggregation step.

4 Learning Curve

The key diagnostic: monotonic convergence, exactly as predicted.

The learning curve confirms the Stage 1 diagnosis was correct:

  1. No overshoot. Val AUC increases monotonically from epoch 1 to ~60, then plateaus. Compare Stage 1 where val AUC peaked at epoch 6 then collapsed to 0.62.
  2. Val > Train throughout. The validation set (northern Rwanda, sparser) is an easier classification problem than the training set (includes dense Kigali). This is not overfitting — it’s distribution shift in the favorable direction.
  3. ReduceLROnPlateau worked as designed. Six halvings (epochs 34→60) drove the LR from 5e-6 to 1e-7. Each halving produced a small AUC bump before the next plateau. After epoch 60 (LR = 1e-7, the floor), no further learning occurred and patience ran out at epoch 103.

6 What We Learned

Stage 1 → 1b is a textbook case of learning rate as the binding constraint. The architecture, data, and loss function were all adequate — the model failed to converge only because the optimizer overshot the loss minimum on every update. Three theoretical concepts explain the full picture.

6.1 1. Overshoot ≠ Overfitting

A common misreading of Stage 1’s learning curve would be “the model overfit after epoch 6.” This is wrong. Overfitting produces a signature where training loss keeps improving while validation loss degrades — the model memorizes training examples at the expense of generalization. Stage 1 showed the opposite: both train and val AUC degraded together after epoch 6. This is the signature of optimization instability, not statistical overfitting.

The distinction matters because the remedy is different. Overfitting calls for regularization (dropout, weight decay, data augmentation, early stopping). Optimization instability calls for learning rate reduction — which is exactly what fixed it.

6.2 2. The Loss Landscape Geometry

With 6,849 parameters, unet_tiny defines a loss surface in ~7,000-dimensional space. At LR = 5e-5, each Adam update moved the parameter vector by a step large enough to jump across narrow valleys in this surface. The optimizer found the right valley early (epoch 6, AUC = 0.93) but could not settle into it — each subsequent step bounced the parameters to a worse region.

At LR = 5e-6, the step size is 10x smaller. The optimizer navigates the same landscape but can now descend into narrow valleys without overshooting. The monotonic convergence in Figure 1 confirms this: the optimizer followed a smooth downhill path for 60 epochs before reaching the valley floor.

This connects to a general result in optimization theory: for a loss function with Lipschitz-continuous gradients (smoothness constant \(L\)), convergence is guaranteed only when \(\eta < 2/L\). Stage 1’s LR exceeded this bound; Stage 1b’s did not. We did not need to compute \(L\) explicitly — the learning curve itself is the diagnostic.

6.3 3. Adaptive Scheduling as Coarse-to-Fine Optimization

ReduceLROnPlateau implements a simple but effective coarse-to-fine strategy. At the initial LR (5e-6), the optimizer takes relatively large steps — good for covering distance toward the optimum. Once progress stalls (5 epochs without val AUC improvement), the LR halves, enabling finer adjustments. This repeated six times (5e-6 → 2.5e-6 → … → 1e-7), each halving squeezing out a small additional gain.

The LR schedule table above shows the diminishing returns clearly: the six halvings after epoch 33 collectively added only +0.0013 AUC. This is consistent with the optimizer having already reached a flat region of the loss surface — additional precision in the step size yields negligible improvement. The model has converged.

6.4 4. Probability Calibration: Why the Threshold Shift Matters

The most consequential change is not the AUC improvement but the decision threshold shift from 0.15 to 0.90. In Stage 1, the model produced poorly separated probability distributions — building and non-building pixels overlapped heavily, requiring a very low threshold to capture buildings (at the cost of false positives). In Stage 1b, the distributions are well-separated: buildings cluster near \(p = 1.0\) and non-buildings near \(p = 0.0\).

This has a direct practical consequence for Stage 2. When we aggregate predicted building density per district, we need the raw probability map to be interpretable as a confidence-weighted density estimate. A well-calibrated model (threshold ≈ 0.5–0.9) produces probability maps where pixel values approximate the true probability of a building being present. A poorly calibrated model (threshold = 0.15) produces maps where the raw values are not meaningful — you can threshold them but cannot average them. Stage 1b’s calibration makes the probability map directly usable as input to district-level aggregation.

6.5 5. Val > Train AUC: Distribution Shift, Not a Bug

Throughout training, validation AUC exceeds training AUC (Figure 1). This is unusual but not problematic. The spatial train/val split assigns southern/central Rwanda (rows 0–2, including dense Kigali) to training and northern Rwanda (row 3, sparser rural areas) to validation. Building detection in sparse rural areas is an easier discrimination problem — fewer ambiguous pixels, clearer contrast between buildings and surrounding vegetation. The model trained on the harder distribution (dense urban) generalizes well to the easier distribution (sparse rural), which is the favorable direction for our application.

If anything, this increases confidence in the model’s robustness: it was trained on the most difficult region and still performs well on a geographically distinct area.

7 Possible Improvements vs. Moving to Stage 2

A natural question: should we keep tuning Stage 1 or move forward? Below we evaluate the most promising hyperparameter improvements and argue that Stage 2 is the correct next step.

7.2 Why We Move to Stage 2 Now

Three reasons:

  1. The model has converged. 43 epochs at min_lr with no improvement is strong evidence that unet_tiny has extracted all it can from 3 bands. Further LR tuning will not break through the label ceiling (~0.97–0.98 F1 imposed by Open Buildings accuracy).

  2. The binding constraint has shifted. At AUC = 0.976, the bottleneck is no longer “can the model detect buildings?” — it clearly can. The question is now “does building density correlate with economic activity?” That is a Stage 2 question (district-level validation against census data), not a Stage 1 question.

  3. Architecture improvements run in parallel. The buildingsALL-Rwanda2022 experiment (unet_medium, 10 bands) requires GPU and will run on Quest. This tests whether additional spectral bands (Red Edge, SWIR1, etc.) improve discrimination. But we do not need to wait for it — Stage 2 validation can proceed with the current 3-band model, and if the 10-band model improves, we re-run Stage 2 with the better checkpoint.

7.3 What Would Change This Decision

We would return to Stage 1 tuning if:

  • Stage 2 reveals that AUC = 0.976 is not sufficient for meaningful district-level correlation (e.g., Pearson r < 0.3). This would suggest the model’s pixel-level accuracy is too low for aggregation to work, and further improvement is needed.
  • The 10-band Quest model significantly outperforms 3-band (e.g., AUC > 0.99), suggesting that architecture/data was the bottleneck, not optimization. In that case, further hyperparameter tuning of the larger model would be warranted.

Neither outcome blocks us from starting Stage 2 now.

8 Bug Fix: plot_metric_progress() Crash

The training run crashed at the end during post-training plotting:

KeyError: 'precision'
  at vendor/mlgis-ponds/src/mlgis_helpers/evaluation.py:222

Root cause: plot_metric_progress() unconditionally accessed progress["precision"], progress["recall"], and progress["f1"] — but the _save_final_results() function in training.py only populates progress["auc"] and progress["train_auc"] from the Keras training history. Keras does not compute per-epoch precision/recall by default.

Fix applied: Guarded lines 222–224 with if "key" in progress: checks. The function now plots whichever metrics are available and skips the rest. This crash prevented val_metrics.json and training progress plots from being saved, but the best.keras model checkpoint (saved during training via ModelCheckpoint callback) was unaffected.
